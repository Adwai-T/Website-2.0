PickLists

A noraml picklist set can be promoted to a global picklist by using Promote to Global Value Set button on the picklist.

Global Picklists

Global value sets let you share the same picklist values with more than one picklist field.

Global value sets are always restricted and cannot be converted into unrestricted, this protects the values as changing global value set changes it for all the fields that use it.

Quick search option Picklist Value Sets.

We can add new value, reorder, replace, and change the default value of a global value set. We can also edit, delete, and deactivate individual values.

When custom picklist use a global picklist set, a single select can be changed to multi select, but it cannot be changed to any different datatype.

Custom picklist field's association with a global value set cannot be undone.

Dependent Picklist

When there are two picklist or a picklist and a checkbox field, a dependency can be created between them by clicking the Field Dependencies button on the top right of the Fields and Relationship page an object.

Then select Controlling Field and Dependent Field. Next choose the options from dependent picklist values to be included when a value from the controlling picklist is selected and click Include Values, then save.

A controlling picklist can at most have 300 values.

Picklist Limits

For standard picklist the value can be 255 character long, which count the line break and return character.

Restricted picklist can have upto 1000 values that includes active as well as inactive.

Unrestricted picklist can have upto 1000 values and upto 4000 inactive values.

In custom multiselect picklist a user can have upto 500 values, each with upto 255 characters and can select upto 100 values at a time on a record.

Global picklist set can have a combination of 1000 active and inactive values.

Upto 500 Global picklist set can be created in an org, and any number of picklists can use these sets.

If a global picklist value set is applied to more than 13 different object, the values can be deactivated but not replaced or deleted.

Controlling picklist can have atmost 300 values.

Relationships

Relationship between Account and Opportunity

Relationship between Account and opportunity is a standard relationship even if shows as a lookup relation ship in schema builder.

This is a kind of default relationship between the two objects such that we can create a rollup summary on the account object for opportunity.

Also when the account is deleted, the opportunity related to the account are deleted.

Many To Many Relationship

Salesforce does not provide a direct option to create a many to many realtionship.

However a many to many relationship can be created by using two Master Detail Relationship on a Juntion object that has the two object that need to be linked with many to many relationship as the Master.

The First Master Details relationship created on the Junction object becomes the Primary relationship and effect the records of the junction object in following ways -

  • Look and feel of edit page, icons and color of primary relationship are used.
  • Record ownership is inherited from the Owner field value of the primary Master, even if there is no Owner field on details object record.
  • Division - if used is inherit their division from primary master record.

For Related List Label use an appropriate name that represent the relationship between the two objects.

If the primary master-detail relationship is deleted or converted to lookup, the secondary master detail relationship becomes primary.

Customize the Related List

To create a seemless experience for user, the related list where the relationship is displayed can be customized.

Edit the page layout for each of the Object linked by the Many to Many relationship. In the related list click on the wrench icon and add the field that refers to the other object.

This will make it convenient for the user to see the related object directly in the related list.

Process Automation

Salesforce Flow

Salesforce Flow provides declarative process automation.

  • SalesForce Flow -> Product
  • Flow Builder and Process Builder -> Tools

There are other automation tools that can be used such as workflow rules and process builder that are used but are going to be retired and hence it is important to use Salesforce flow.

  • Use Cases for Salesforce Flow :
  1. Set up automated tasks and processes.
  2. Create a guided tutorial or wizard with screens.
  3. Connect to external systems.
  4. Add automation to your pages and apps.
  5. Reuse what you build.
It is always best to start with declarative non code tools first and work our way up to code solutions as they are much easier to manage.
  • Cases and what automation tool to use :
  1. Guided visual experience : processes that need input from users -> Flow Builder
  2. Behind-the-scenes automation : user input isnâ€™t needed. and salesforce has all the data that is needed -> Flow builder or process builder (if existing) or Apex
  3. Approval automation : Business processes that determine how a record, like a time-off request, gets approved by the right stakeholders. -> Approvals
  • Use of Flow Builder :
  1. Automate a guided visual experience.
  2. Start a behind-the-scenes business process: such as create, update, delete a record or when platform event occur or even at a specified time or frequency.

Use of Process builder :

It is now not recommended to use process builder and instead use flow builder as they have the same functionality and flow builder is the current tool and will be updated to get better.

  • Use Apex With flow :

Use Apex when you need more functionality than is available in Flow Builder. Build the more complex functionality as invocable Apex methods. Then call the resulting Apex as an Apex action in the process or as an Apex action element in the flow.

Process Builder

Official Docs Process Builder.

Automate if/then business processes and see a graphical representation of your process as you build.

Use Process Builder in Quick Find.

Component of Process :

  • Trigger : Identify When the Process Should Run.

1. Only when record is created. 2. Anytime a record is created and edited.

  • Criteria : Determine Whether or Not to Execute Actions.

1. Set filter conditions. 2. Enter a custom formula. Like in validation rules, the formula must resolve to true or false.

3. Opt out of criteria and always execute the associated actions.

  • Actions : What the Process Should Do.

1. Immediate Action. 2. Scheduled Action.

  • Actions that can be performed

1. Create records. 2. Update the record that started the process or any related record.

3. Submit that record for approval.

4. Update one or more related records.

5. Send emails using a specified email template.

6. Post to a Chatter feed.

Process Types

  1. Record Change : A record is created or edited
  2. Invocable : Itâ€™s called by another process
  3. Platform Event : A platform event message is received

Flow Builder

Official Docs : Flow

Salesforce Flow : Encompasses building, managing, and running flows and processes.

Flow is an application that automates a business process by collecting data and doing something in your Salesforce org or an external system.

Access Flows by using Flows in quick find.

Flow Builder : a point-and-click tool for building flows.

Components of Flow builder

  1. Elements : Screen, Logic, Actions, Integrations.
  2. Connectors : define the path that the flow takes at run time.
  3. Resources : are containers that represent a given value, such as field values or formulas.

Keyboard ShortCuts in flow builder

  • To zoom in, press CTRL+=.
  • To zoom out, press CTRL+-.
  • To zoom to fit, press CTRL+Alt+1.
  • To zoom to view, press CTRL+Alt+0.
Only flow admins (users with the Manage Flow user permission) can run inactive flows.

Users

Users use Salesforce licenses. Each user has exactly one license and one profile.

Available Licenses for the organization can be viewed in Company Information.

Every Salesforce user is identified by a username, a password, and a single profile.

Together with other settings, the profile determines what tasks users can perform, what data they see, and what they can do with the data.

Profiles

Salesforce has an enhanced profile user interface that makes it easy to find and modify profile settings. User Management Settings | (Enable) Enhanced Profile User Interface.

If a user already is given access to some data in the profile that permission cannot be taken away by any other security setting like permission set, so the profile should be the most restrictive and access should be opened up using permission set and sharing settings.

  • Profiles control the access to object and fields.
  • They also control access to Tabs and Apps.
  • Login Hours can also be set for particular profiles.

Standard Profiles

They can be views as templates that can be cloned to create custom profiles that are used. They can be assigned directly to a user but have limitations over custom profiles that are created by cloning them.

Standard profiles permission cannot be edited but some settings can be customized, such as

  • Custom App Settings
  • Tab Settings
  • Desktop Integration Client Options
  • Session Settings
  • Password Policies

Custom Profiles

Custom Profiles are created by using standard profiles as template and they are mostly assigned to users. A Custom profile cannot be created from scratch, a standard profile that is the closest to the functionality need to be used and cloned.

Unlike Standard Profiles all the settings in Custom Profiles can be modified.

Permission Set

The interface for configuring permission sets look very similar to that of the profile as they have the same type of permissions that can be granted.

Permission sets details page Manage Assignments (button) can be used to see all users assgined the permission set and add that permission set to new users.

Also on User Details page the permission sets assigned to the user can be viewed and new permission sets can be assigned.

Permission Set Group

Group a number of permission sets that might to be given to simialr type of users.

In this a newer feature of permission set group is mute permissions sets in group in which some permissions sets can be muted and reactivated/unmuted as needed.

We can also see under combined permission, all the permissions together that are granted by all the permission sets in the permission set group.

Objects

By setting permissions on a particular type of object, you can prevent a group of users from creating, viewing, editing, or deleting any records of that object.

Record detail pages can be reached by placing a record ID at the root of the URL.

You can set object permissions with profiles or permission sets. Permission sets grant additional permissions and access settings to a user.

Users are assigend one profile and can be assigned any number of permission sets.

Permissions can also allow user to create or edit records of a given type, run reports, and customize the app.

The View All Data and Modify All Data overrides all other sharing settings, role hierarcy, and sharing rules, so should only be assigned to system administrator.

Fields

You can restrict access to certain fields, even if a user has access to the object.

Like objects field access can be controlled by the profile and by using permission sets.

Field Access are given just below object settings, and have two options (Read Access, Edit Access).

In field details page in object manager, View Fields Accessibility (button top center) shows the accessibility for different profiles. Where it is used is another button next ot view fields accessibility that shows where the field is referenced (eg page layouts). Where it is used is not availabe for standard fields.

There is a 900 maximum hard limit on total custom fields per object.

Records

You can allow particular users to view an object, but then restrict the individual object records they're allowed to see.

Controlled inside Sharing Settings.

Organization Wide Default is the most restrictive, as you go down the list the options can be used to open up the organization security for more users selectively.

Organization Wide Default

Specify the default level of access users have to each otherâ€™s records.

In Organization Wide Default all standard objects have Grant Access Using Hierarchies enabled by default and it cannot be changed while it can be disabled for custom objects.

The options for each object depend on the object and can be Public Read/Write, Public Read Only, Private, Public Read/Write/Transfer (for objects such as Cases, Leads), Controlled By Parent(In case of details object in a Master-Details Relationship).

Role Hirerachy

Role Hierarchy provides vertical access to records. The user above in the hierarchy has access to all the records of the user with roles below.

Not all user need to be added to the role hierarcy, some can be outside of the role hierarcy and their records will not be shared by the role hierarchy.

Sharing Rules

Use Sharing Rules to make automatic exceptions to the organization wide sharing settings for defined set of users.

The sharing can be done with Roles, Roles and Subordinates or Public groups.

Then access can be given to object themselves and the related objects.

Users' Identity

Multifactor Authentication

When MFA is turned on, users need to provide multifactor every time they log in.

Orgs can be setup to require MFA for additional circumstances such as

  1. When user access Salesforce APIs.
  2. When user access a connected app, dashboard, or report. This process is known as step-up or high Assurance authentication.
  3. During a custom login flow or within a custom app.

In multifactor authentication the user provides two evidences :

  1. Something that the user knows like username and password.
  2. Something that is in users possession like mobile device with an authentication app or a physical security key. It can be done through Salesforce Authentication(Mobile Application), ThirdParty TOTP(Time based One-time Password) (Google Authenticator, Microsoft Authenticator, Authy etc), Security Keys which are small physical token that look like thumb drive (compatible with FIDO universal Second Factor, or FIDO2 WebAuthn Standards), or Build-in Authenticators like fingerprint or facial recofnition scanner that are build into user's device.

Enable Multifactor Authentication

Before setting up MFA setup Session Settings so that Multi-Factor Authentication is in the High Assurance Category as it can prevent You and other Admins from loggin in.

Enable MFA for Select Users

Enable MFA for select users by assigning the Multi-Factor Authentication for User Interface Login user permission. This can be done by editing profiles or by creating a permission set that can be then assigned to specific users.

Steps :

  1. Create a permission set and name it appropriately.
  2. Once created, go to the permission set details page.
  3. Under System | System Permissions (Click link) | Edit (button) | Multi-Factor Authentication for User Interface Logins (check box) | Save.
  4. Multi-Factor Authentication for API Logins option is also available here.
  5. Assign this permission set to the users as needed.
  6. On details page of new permission set, click Manage Assignments.
  7. Click Add Assignment On the list of users, select the checkbox next to the users.
  8. Click Assign | Done.
There are multiple options Named Multi-Factor Authentication, check the Multi-Factor Authentication for User Interface Logins.

Turn On MFA for Everyone in Organization

Setup | Identity | Identity Verification | Select Require multi-factor Authentication for all direct UI logins to your Salesforce org (checkbox) | Save.

After the setting is enabled everyone in the org, regardless of whether they were previouly assigned the MFA user permission, is now required to complete MFA when logging in.

Sandbox

A Salesforce sandbox is a place for the users to test and build without risk of changing or losing the valuable data on the production organization. Sandboxes help user to keep production organizations data clean during training, development and testing.

Sandboxes are accessed through https://test.salesforce.com. Use username with a period and the name of the sandbox at the end like name@example.com.SanboxName and password on the day that sandbox was created.

Types of Sandboxes

Developer Sandbox has all of the production configuration but none of the production data. They can be refreshed or pull in latest configuration from production once a day.

Production Configuration included in sandbox are Configurations, Apex, Metadata and All Users.

Has capacity of 200MB.

Good for development and testing.

Developer Pro Sandbox has all the production configuration but none of the production data. They can hold more data than the Developer Sandbox and can be refreshed once per day.

Has capacity of 1 GB.

Good for development and quality assurance tasks, testing, and user training.

Partial Copy sandbox has all the production configuration and also a sample of the real-world data from the production organization what was defined using the sandbox template. They can be refreshed once every 5 days.

Has capacity of 5GB.

Good for user acceptance testing, integration testing, and training, because it contains some of your production data.

Full Copy sandbox is a full copy of the production organization. It includes all the configuration and most if not all data. Can be refreshed every 29 days.

Capacity is same as production environment.

An ideal testing environment since it is just like production. This is the only type of sandbox that supports performance testing, load testing, and staging. You probably donâ€™t want to use Full sandboxes for development, though, because you can only refresh configurations and data every 29 days, and that refresh can take days to complete.

The type and quantity of sandboxes you can set up and use depends on your license.

Create a New Sandbox

A sandbox needs to be set up in Salsforce production. It cannot be setup in a developer or trailhead playground.

Setup | Sandbox (Quick Find box) | Sandbox | New Sandbox | Name Sandbox | Select create from dropdown (mostly production) | Select type of sandbox to create | Create.

Sandboxes once created will take several minutes or even days in the case of Full Copy to be ready to use. A notification email will be sent once the sandbox is ready to use.

Once sandbox is created, check email deliverability settings (Setup | Deliverability (Quick Search) | System email only or No access). By default the System email only is the deliverability access level on any new Sandbox, but still should be checked to make sure.

Refresh Sandbox

To add new configuration and data from production to sandbox, they need to be refreshed by going to Sandboxes list in Setup | Sanbox and Refresh next to the sandbox's name.

It is important to note that a refresh will destroy any of the data or customizations created in that sandbox since last refresh.

Also after refresh the email Deliverability must be check again.

  • Dashborads style apps.
  • Point app to solve particular task.
  • Custom record pages for objects.
  • Custom home pages.
Structure of Lightning pages adapt for devices its viewded on.

Lightning Page Types

  • App Page
  • Home Page
  • Record Page
When a page is Edited for the first time, salesforce makes a copy of the standard page. The copy is what then can be edited in Lightning App Builder. The page created like this, as a copy, retain a reference to the standard page, which will then be used by salesforce when new features need to be added. If a page is created from scratch, the new page capabalities Salesforce rolls out don't appears automatically.

To Create a new Page go to - Setup | User Interface | Lightning App Builder. Here click new and then select the type of page that needs to be created.

To Deactivate A page click Activation, click the App | Page Type |Profile tab | Remove Assignments.

Lightning Components

It is a compact, configurable and reusable element that can be added to Lightning App Builder.

  • Standard Components - Lightning component build by salesforce.
  • Custom Components - Made with Aura or LWC.
  • Thrid party Components on AppExchange
